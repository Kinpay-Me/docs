## Services used

| Service | Purpose |
|---|---|
| ECS Fargate | Runs the backend API containers (serverless containers — no EC2 to manage) |
| RDS (PostgreSQL) | Managed PostgreSQL database |
| S3 | File storage — avatars, signatures, repayment proof, contract PDFs |
| ALB | Application Load Balancer — routes traffic to ECS tasks, terminates TLS |
| ACM | Manages TLS certificates for the API and frontend domains |
| Route 53 | DNS for `api.kinpay.me`, `app.kinpay.me`, `dashboard.kinpay.me` |
| ECR | Container registry — Docker images pushed here, pulled by ECS |
| CloudWatch | Logs and metrics from ECS tasks |

## Environment variables (AWS)

| Variable | Description |
|---|---|
| `AWS_ACCESS_KEY_ID` | IAM credentials for S3 access |
| `AWS_SECRET_ACCESS_KEY` | |
| `AWS_REGION` | e.g. `eu-west-1` |
| `S3_BUCKET_NAME` | Bucket for all file uploads |
| `S3_REGION` | Usually same as `AWS_REGION` |

## S3 file structure

```
{bucket}/
  avatars/{user_id}/avatar.{ext}
  signatures/{user_id}/signature.{ext}
  contracts/{contract_id}/contract.pdf
  repayments/{repayment_id}/proof.{ext}
```

All files are private. Access is granted via pre-signed URLs generated by the backend (15-minute expiry). See [[link:data-architecture/contracts-entity|Contracts]] and [[link:data-architecture/payments-entity|Payments]] for the corresponding database references.

## ECS task configuration

The backend runs as a Fargate task with:
- **CPU:** 512 vCPU units
- **Memory:** 1024 MiB
- **Min tasks:** 1 (always-on)
- **Max tasks:** 10 (auto-scaled on CPU utilisation)

The task definition pulls the image from ECR and injects environment variables from AWS Secrets Manager. For the full variable reference, see [[link:deployment/environment|Environment Variables]].

## RDS configuration

- **Engine:** PostgreSQL 15
- **Instance class:** `db.t3.small` (production)
- **Multi-AZ:** Enabled in production
- **Automated backups:** 7-day retention
- **Database name:** `kinpay`

## Deployment

CI/CD pushes a new Docker image to ECR and triggers an ECS rolling deployment. The old task keeps serving traffic until the new task passes its health check on `GET /health`.
